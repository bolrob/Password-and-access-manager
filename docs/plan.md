# План разработки

## Цель проекта  
Создать одностраничное Angular-приложение для безопасного хранения учётных данных с генерацией паролей.  
Пользователь через личный кабинет управляет своими записями (логин/пароль/URL), категоризирует их,  
получает оценку надёжности, отслеживает историю изменений и планирует замену устаревших паролей.  
Данные хранятся локально (mock-сервер + localStorage для токена), авторизация разделяет доступы.

## Пользовательские сценарии  
1. **Регистрация и вход** – новый пользователь создаёт аккаунт, входит в систему и видит пустой vault.  
2. **Добавление новой записи** – заполнение формы с автоматической генерацией надёжного пароля,  
   проверкой сложности, выбором категории и установкой напоминания о смене.  
3. **Поиск и управление** – быстрый поиск по названию/тегам, фильтр по категориям,  
   сортировка по дате/надёжности, пометка устаревших паролей, просмотр истории изменений.  
4. **Экспорт данных** – экспорт всех записей в зашифрованном JSON/CSV (защита мастер-паролем).

## Этапы разработки  
| Срок | Задача |
|------|--------|
| До конца апреля | Базовая структура проекта, роутинг (ленивые модули), auth-сервис (логин/регистрация через mock-API), guards, компонент vault (список записей без CRUD), форма с валидацией (генерация пароля как отдельный компонент), разделение на features/shared/core. |
| До конца мая | Полный CRUD записей (добавление/редактирование/удаление), поиск/фильтр/сортировка, расчёт надёжности, напоминания, история изменений, экспорт. State management на Signal Store. Interceptors. Unit-тесты (Jest, ≥15), компонентные тесты (Playwright, 3 сценария). Настройка ESLint/Stylelint/Prettier, CI/CD (GitLab CI или GitHub Actions) с деплоем на Vercel/Firebase. Lighthouse ≥80. |

## Риски  
1. **Неверный выбор структуры состояния** — снижение: использовать Signal Store с чёткими фича-слайсами.  
2. **Сложность с mock-сервером для истории и напоминаний** — снижение: реализовать простой flat-file mock на JSON Server или in-memory-data-service с расширенными возможностями.  
3. **Проблемы с производительностью при большом числе записей** — снижение: OnPush, trackBy, виртуальный скролл (Taiga UI Scroller).  
4. **Отсутствие дизайн-макета тормозит вёрстку** — снижение: сразу сделать прототип в Figma/от руки на 4 экрана.  
5. **Недостаток времени на полное тестирование** — снижение: писать тесты параллельно с фичами, минимальный сценарий покрыть сразу.
